#### 代码绘制规范

```typescript
// 所有代码绘制的精灵统一使用 PixelRenderer 类
// 像素数据用二维颜色数组定义，支持运行时变色

interface PixelSprite {
    width: number;        // 精灵宽度（像素）
    height: number;       // 精灵高度（像素）
    pixels: string[][];   // 像素颜色数组 ['#ff0000', '#00ff00', ...]
    transparent: string;  // 透明色标记（默认 '#00000000'）
    palette?: string[];   // 可替换的调色板（用于元素变色）
    frames?: PixelSprite[][]; // 动画帧数组
}

// 元素变色示例：同一个精灵，传入不同调色板即可变色
const baseSprite = loadSprite('babo_warrior');
const fireSprite  = applyPalette(baseSprite, FIRE_PALETTE);   // 火焰色
const iceSprite   = applyPalette(baseSprite, ICE_PALETTE);    // 冰蓝色
const thunderSprite = applyPalette(baseSprite, THUNDER_PALETTE); // 雷电黄
```

#### AI 生成规范

- 使用 GenerateImage 工具生成
- 提示词必须包含：`pixel art, game sprite sheet, transparent background, retro game style, clean pixel edges`
- 尺寸必须指定：角色 32×32、敌人 16×16、Boss 64-128、图标 32×32
- 生成后保存到 `/workspace/babo-survivors/public/assets/sprites/` 对应子目录
- AI 生成的图片仅用于静态展示（UI、头像、宣传），不用于战斗中的动态精灵

## 12. agent-browser 浏览器自动化规则

### 12.1 适用场景

agent-browser 技能在本项目中的具体用途：

| 场景 | 用途 | 触发时机 |
|------|------|----------|
| **实时预览游戏** | 启动 dev server 后用浏览器打开，截图查看实际效果 | 每完成一个功能模块 |
| **UI 还原度验证** | 对比设计稿和实际渲染效果，检查像素对齐、布局、颜色 | UI 开发完成后 |
| **移动端模拟测试** | 用浏览器 DevTools 切换到 iPhone 17 Pro Max 竖屏视口，验证响应式 | 移动端适配完成后 |
| **交互测试** | 模拟点击按钮、输入文字、滚动页面，验证交互逻辑 | 功能开发完成后 |
| **性能 profiling** | 通过浏览器 Performance 面板检查 FPS、内存、渲染瓶颈 | 性能优化阶段 |
| **Bug 复现** | 按照用户描述的操作步骤在浏览器中复现问题 | 收到 Bug 反馈时 |
| **截图存档** | 每个里程碑截图保存到 `/workspace/B-BABO-game/docs/screenshots/` | 阶段性验收 |

### 12.2 使用流程

```
1. 启动开发服务器（npm run dev，blocking=false）
2. 等待服务器就绪（browser_wait_for）
3. 用 browser_navigate 打开游戏页面
4. 用 browser_take_screenshot 截图查看效果
5. 如需交互：browser_snapshot → browser_click/browser_type
6. 验证完毕后截图存档
```

### 12.3 注意事项

- dev server 使用 `blocking: false` 启动，不阻塞后续操作
- 截图保存到 `docs/screenshots/` 目录，按日期和功能命名
- 移动端测试时设置视口为 `432×932`（iPhone 17 Pro Max 逻辑分辨率）
- 每次截图前先 `browser_wait_for` 确保页面加载完成

## 13. 子代理并行开发规则

### 13.1 核心原则

**当多个开发任务相互独立时，必须使用子代理并行开发，不要串行逐个完成。**

### 13.2 并行开发策略

```
主代理（我）的职责：
├── 任务分解 → 将大任务拆分为独立子任务
├── 分发任务 → 启动子代理并行执行
├── 审核核查 → 子代理完成后审查代码质量
├── 集成合并 → 解决冲突，合并代码
└── 推送 GitHub → 最终验证后推送
```

### 13.3 可并行的任务类型

| 任务组合 | 并行方式 | 说明 |
|----------|----------|------|
| 多个独立角色数据 | 3 个 Explore 子代理同时编写 | 角色间无代码依赖 |
| 多个独立武器数据 | 3 个 Explore 子代理同时编写 | 武器间无代码依赖 |
| 多个独立敌人数据 | 3 个 Explore 子代理同时编写 | 敌人间无代码依赖 |
| UI 界面 + 游戏逻辑 | 1 个 UI 子代理 + 1 个逻辑子代理 | UI 和逻辑可同时开发 |
| 前端代码 + 测试用例 | 1 个开发子代理 + 1 个测试子代理 | TDD 模式 |
| 文档更新 + 代码开发 | 1 个文档子代理 + 1 个代码子代理 | 文档和代码独立 |
| 多个地图瓦片数据 | 3 个 Explore 子代理同时编写 | 地图间无依赖 |
| 音效系统 + 粒子系统 | 2 个子代理同时开发 | 系统间独立 |

### 13.4 不可并行的任务

| 任务 | 原因 | 必须串行 |
|------|------|----------|
| ECS 核心框架 | 所有系统依赖它 | 先完成框架 |
| 游戏主循环 | 所有场景依赖它 | 先完成循环 |
| 渲染管线 | 所有视觉依赖它 | 先完成渲染 |
| 输入系统 | 所有交互依赖它 | 先完成输入 |
| 碰撞系统 | 战斗系统依赖它 | 先完成碰撞 |

### 13.5 子代理使用规范

```typescript
// 并行开发示例：同时开发 3 个独立角色
// 在一条消息中同时发出 3 个 Task 调用

// 子代理 1：开发 B-BABO战士 + B-BABO法师
Task("开发战士和法师角色数据", subagent_type="general_purpose_task")
```
